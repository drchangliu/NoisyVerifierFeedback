How much should each LLM trust its static analyzer — and how should you filter its findings?

AI coding agents fix static-analyzer findings in a loop, but most findings are false alarms (combined Semgrep+Bandit precision is 23.8% on LLM-generated Python). The answer is neither to trust every finding nor to ignore them all, but to filter per model. From real fix-loop interactions we measure two rates and derive one threshold that tells you which findings a given model should even see.

Fix rate q q = (true findings fixed) / (true findings shown)
Of real findings surfaced to the model, the fraction it correctly fixes (secure and tests still pass). Higher is better.
Regression rate r r = (working code broken) / (false alarms shown)
Of false findings surfaced, the fraction where the model’s “fix” breaks working code. Lower is better — this is the ranking key. A low-r model recognizes false alarms instead of obeying them.
Surfacing threshold τ* τ* = r / (q + r)
Surface a finding to this model only if its rule’s historical precision p exceeds τ*; filter everything below. Low r ⇒ low τ* ⇒ surface almost everything; high r ⇒ high τ* ⇒ filter aggressively.

In the table below, models are ranked by r (lowest first). Read across: a small τ* means the model can safely see almost all findings; a large one means most should be filtered out. Better fixed policy names which of the two simple defaults — naive (surface everything) or selective (filter at 50% precision) — lands closer to that model’s τ*. More on the table columns (the three rates q, r, τ* are defined at the top):

  • Better fixed policy — the better of the two deployed fixed policies, naive (surface every finding) vs selective (only rules above 50% precision). “Naive” does not mean surfacing everything is optimal — the optimum filters at the model’s own τ*, which is above zero for every model.
  • Baseline capability (JointPass@0) — fraction of items whose pre-feedback code is both functionally correct and vulnerability-free, on the 51-item core benchmark (combined Semgrep+Bandit analyzer, multi-seed).
  • HumanEval pass@1 — external functional-capability axis (disjoint tasks, no security signal).
  • in-sample (Notes) — one of the five models used to formulate the r-law; all others were measured prospectively.

The headline: r falls steeply with capability (Spearman −0.90) while q stays flat — better models are less gullible — so the right feedback policy is a property of the model and expires with every model update. One frontier model (Claude Fable 5) could not be measured at all: its safety layer refused 9/10 benchmark requests.
