LLM Trust Leaderboard

How much should each LLM trust its static analyzer — and how should you filter its findings?

AI coding agents fix static-analyzer findings in a loop, but most findings are false alarms (combined Semgrep+Bandit precision is 23.8% on LLM-generated Python). The answer is neither to trust every finding nor to ignore them all, but to filter per model — from real fix-loop interactions we measure two rates and derive one threshold that says which findings a given model should even see.

The three rates

Fix rate q higher is better

Of the real findings surfaced to the model, the fraction it correctly fixes — secure and tests still pass.

q = true findings the model fixed true findings shown

Regression rate r the ranking key lower is better

Of the false findings surfaced, the fraction where the model’s “fix” breaks working code. A low-r model recognizes false alarms instead of obeying them.

r = working code the “fix” broke false alarms shown

Surfacing threshold τ*

Feed the model a finding only if its rule’s historical precision p exceeds τ*; filter everything below. Low r ⇒ low threshold ⇒ surface almost everything; high r ⇒ filter aggressively.

τ* = r q + r
Surfacing rule: surface a finding  ⇔  p > τ*

More on the columns
  • Params — total parameters where the developer discloses them (MoE = mixture-of-experts, where far fewer are active per token); “—” = undisclosed, as for most frontier API models. Note: capability, not size, predicts r — the x-axis (JointPass@0) and HumanEval are the capability measures, and several small models resist false feedback better than larger ones.
  • Better fixed policy — the better of the two deployed fixed policies, naive (surface every finding) vs selective (only rules above 50% precision). “Naive” does not mean surfacing everything is optimal — the optimum filters at the model’s own τ*, which is above zero for every model.
  • Baseline capability (JointPass@0) — fraction of items whose pre-feedback code is both functionally correct and vulnerability-free, on the 51-item core benchmark (combined Semgrep+Bandit analyzer, multi-seed).
  • HumanEval pass@1 — external functional-capability axis (disjoint tasks, no security signal).
  • in-sample (Notes) — one of the five models used to formulate the r-law; all others were measured prospectively.

The headline: r falls steeply with capability (Spearman −0.90) while q stays flat — better models are less gullible — so the right feedback policy is a property of the model and expires with every model update. One frontier model (Claude Fable 5) could not be measured at all: its safety layer refused 9/10 benchmark requests.
